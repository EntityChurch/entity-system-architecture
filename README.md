# Entity System Architecture

**The optional capability layer above the core protocol.** This repo holds everything that is
*not* the irreducible protocol: the extension family, the SDK conventions, the application-layer
(L5) conventions, and the developer guides. The model is **"agree with us → you interoperate"** —
each extension is a contract you opt into; divergence is allowed, convergence is offered.

The mandatory protocol floor lives separately in the `entity-core-protocol`
repository; the publishable tooling in `entity-system-arch-tools`.

> **Convergence floor:** this surface (`--profile full`) is 3-way converged across the Go, Rust,
> and Python reference implementations at 0-FAIL, and generated by the keystone cohort.

## What's here

- **Extensions** (`EXTENSION-*`) — the capability library. The v1 set (Content, Type, Revision,
  Subscription, Continuation, Inbox, History, Query, Compute, Group, Identity, Attestation,
  Quorum, Clock, Role) is mature; the network/resolution family (Network, Relay, Route,
  Substitute, Registry, Discovery, Encryption) is converging. See the roadmap for stages.
- **SDK** (`SDK-*`) — cross-impl binding conventions over the type surface.
- **Applications** (`applications/`) — L5 content-format conventions (Embed, Semantic Content
  Site) that let web / Godot / terminal front-ends render the same bytes.
- **Guides** (`GUIDE-*`) — developer-facing how-to and discipline docs, including
  `GUIDE-CONFORMANCE` (the conformance methodology + vector index) and
  `GUIDE-EXTENSION-DEVELOPMENT`.

## Roadmaps

Three living, canonical roadmaps — the content/papers projects pull from these:
- **`ROADMAP-EXTENSIONS.md`** — all extensions by stage (the v1 set / the network frontier /
  the forward-feature backlog).
- **`ROADMAP-SDK.md`** — the binding layer.
- **`ROADMAP-APPLICATIONS.md`** — the L5 conventions (be precise: SITE read ships at preview;
  SPACES/REPOS are future/undesigned).

Each domain's M0–M6 maturity ladder lives in its roadmap above. **Note:** a spec's own
`Status:` header is not a reliable maturity signal (many converged extensions still read
"Draft") — use the M-level in the roadmap.

## Relationship to core

Extensions cite core protocol sections. Post-split those are cross-repo references (form: see the
citation convention). The one core→here dependency: the `--profile core` floor exercises
`EXTENSION-TREE` get/put, which ships in core; the rest of TREE's feature set and all other
extensions are here.
