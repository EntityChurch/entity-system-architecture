# Extensions вЂ” Domain Roadmap (canonical, living)

**Target repo:** `entity-system-architecture` (the extension family).
**What this domain is:** the **optional, opt-in capability layer** above the core protocol. An
extension is a contract: implement it and you interoperate with peers that also implement it.
The core floor never requires any of these (except TREE's get/put, which lives in core) вЂ” they
are the "agree with us в†’ you interoperate" surface, validated 3-way (Go/Rust/Python) and
generated by keystone.

> **Canonical & living.** Updated as extensions move; the content/papers projects pull from it.
> Maturity levels (M0вЂ“M6), per-extension source-of-record, and the forward-feature backlog
> (Band E) are defined in **`STATUS-RELEASE-SURFACE-AND-MATURITY-CANONICAL.md`** вЂ” this doc is
> the forward-looking, stage/phase view over that data.

**Floor:** Go/Rust/Python `--profile full` 0-FAIL (release-green) В· 23 extension specs landed + ~10 in proposal.

> **`network-peer-` vs `standard-peer-` is an organizational tag, not structure.** We file
> extensions in those two folders for navigation, but the split carries no boundary meaning вЂ”
> **all 23 extensions ship to `entity-system-architecture`** regardless of folder. The real
> dependency structure is the topology (`TOPOLOGY-CORPUS-FOR-PAPERS.md`): a foundational tier
> (CONTINUATION/SUBSCRIPTION/REVISION/INBOX/HISTORY/QUERY) heavily depended-on, an identity
> sub-cluster (IDENTITY/ATTESTATION/QUORUM/ROLE/GROUP), and a leaf network frontier вЂ” which cuts
> across the folder tags. The exhaustive file list is `INVENTORY-RELEASE-CORPUS.md`.
